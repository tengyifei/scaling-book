Training LLaMA 3 on TPUs

Part 6 of How To Scale Your Model (Part 5: Training | Part 7: Inference)

Let's take a close look at how we'd train LLaMA 3 models on TPU v5p using what we've learned in the previous section. How big are they? How expensive is training in different configurations? How are they sharded? Let's work through some back-of-the-envelope estimates for how the previous sections map onto real models.

What does LLaMA 3 look like?

The LLaMA-3 model family includes 3 main models: LLaMA 3 8B, 70B, and 405B. We‚Äôll mostly focus on 70B, and leave 8B and 405B for you to explore in the problem section at the end. Here‚Äôs the architecture for LLaMA 3-70B, taken from the LLaMA HuggingFace page.

hyperparam value
\(n_\text{layers}\) (L) 80
\(d_\text{model}\) (D) 8,192
\(d_{ff}\) (F) 28,672
\(n_\text{heads}\) (N) 64
\(n_\text{kv_heads}\) (K) 8
\(d_\text{qkv}\) (H) 128
\(n_\text{embeddings}\) (V) 128,256

Counting parameters and FLOPs

Question: From this table, can we calculate the LLaMA 3-70B parameter count? ü§´ Let‚Äôs apply the content of Section 4 and see if we can get 70B!

param formula count
FFW params d_model * d_ff * 3 (for gelu + out-projection) * n_layers 8,192 * 8,192 * 3.5 * 3 * 80 = 56.3e9
Vocab params 2 (input and output embeddings) * n_embeddings * d_model 2 * 128,256 * 8,192 = 2.1e9
Attention params n_layers * [ 2 (for q embedding and concatenated output projection) * d_model * n_heads * d_qkv + 2 (for k and v) * d_model * n_kv_heads * d_qkv] 80 * (2 * 8,192 * 64 * 128 + 2 * 8,192 * 8 * 128) = 12e9
¬† ¬† 56.3e9 + 2.1e9 + 12e9 = 70.4e9

That‚Äôs great! We get the number we expect. You‚Äôll notice as expected that the FFW parameters totally dominate the overall parameter count, although attention is non-trivial.

Takeaway: The 3 big weight matrices in the MLP block are so much larger than all the other arrays in the Transformer that we can typically almost ignore all other parameters when reasoning about model memory or FLOPs. For LLaMA 3-70B, they represent 56B of 70B parameters.

Let‚Äôs look at FLOPs now! Remember the general rules for training from Section 4.

Question: How many FLOPs does LLaMA-3 perform per token per training step? This helps us determine how expensive the whole training process will be.

Click here for the answer, once you‚Äôve thought about it!

Answer: As shown in Section 4, we do roughly \(6 \cdot \text{param count}\) FLOPs per token, so here that‚Äôs roughly 6 * 70e9 = 4.2e11 FLOPs / token. That‚Äôs about half a TFLOP per token per step. Assuming we‚Äôre compute-bound, this should take roughly 4.2e11 / 4.59E+14 = 1ms on a single TPU v5p chip, assuming perfect FLOPs utilization.

Question: LLaMA 3 was trained for about 15 trillion tokens. How many FLOPs is that total?

Click here for the answer, once you‚Äôve thought about it!

Answer: That‚Äôs easy, it‚Äôs just 4.2e11 * 15e12 = 6.3e24 FLOPs total. 6.3 yottaFLOPs. That‚Äôs a lot! On a single TPU this would take 6.3e24 / 4.59E+14 = 435 years. That‚Äôs also a lot!

Question: Let‚Äôs say we wanted to train on a full TPU v5p pod with 16x20x28 = 8960 chips. How long would this take to train at 40% MFU in bfloat16, assuming we are compute-bound?

Click here for the answer, once you‚Äôve thought about it!

Answer: We know that each TPU v5p can perform 4.59e14 FLOPs / second. At 40% MFU, this will take about T = 6.3e24 / (8960 * 4.59e14 * 0.4) = 3.8e6 seconds. This is about 44 days! That‚Äôs fairly reasonable, assuming we can actually achieve 40% MFU.

Question: LLaMA 3-70B was pretrained with a batch size of about 4M tokens. How many TPUs do we need at minimum to train with this batch size? You can assume bfloat16 parameters and float32 optimizer state, and that you checkpoint gradients 4 times per layer.

Click here for the answer, once you‚Äôve thought about it!

Answer: This question is primarily asking about memory usage, since that‚Äôs the only strict constraint on available compute. During training, we have three primary uses of HBM: model parameters, optimizer state, and gradient checkpoints. If we assume bfloat16 weights, float32 optimizer state, and a very conservative gradient checkpointing scheme (3 times per layer), we have:

Params 2 * 70GB ~140GB
Optimizer State 8 * 70GB ~560GB
Gradient Checkpoints 2 * 8192 * 4e6 * 3 * 80 ~20.9TB
Total ¬† ~21.6TB

The total here is about 21.6TB. You notice that gradient checkpointing strongly dominates the memory picture, even with a very conservative checkpointing scheme. We could technically go to 1 checkpoint per layer, or do microbatching, but this is a reasonable picture. With these assumptions, since each TPU v5p has 96GB of HBM, we need 21.6e12 / 96e9 = 225 TPUs. That‚Äôs not very much actually!

Why wouldn‚Äôt we do this? Well, because it would take us 44 days * 8960 / 171 = 1752 days to train. That‚Äôs 6 and a half years. That‚Äôs a lot. Still, this makes it clear that we‚Äôre using these large clusters not because we‚Äôre bound by memory but rather because we need the extra FLOPs.

Question: Under the same assumptions as the question above, if we use 8960 TPU v5p chips, how much memory will we use per-chip?

Click here for the answer, once you‚Äôve thought about it!

Answer: Our total memory is still about 21.6TB, so per-chip we‚Äôll be using about 2.4GB per chip, which is bascially nothing. If we did much more aggressive checkpointing, e.g. 12 checkpoints per layer, we‚Äôd still only be at 8GB per chip. We‚Äôre nowhere near being memory bound during training at these scales.

Takeaways: It is technically possible to train even very large models on very small topologies, with the caveat that they will likely take a long time. Being able to calculate the total FLOPs of a training run allows us to ballpark its training time by assuming a modest MFU and a known topology.

How to shard LLaMA 3-70B for training

Let‚Äôs stick to our setting from above and say we want to train LLaMA 3-70B with 4M token batch size (1024 sequences of length 8192 per batch) on a TPU v5p pod of 8960 chips. Let‚Äôs discuss what the best sharding strategy is for this model.

Question: Under the assumptions above, can we train our model with FSDP alone? To start, let‚Äôs say we can‚Äôt do any sequence/context parallelism. This should be the first idea you have, since it‚Äôs simple and will introduce no extra communication if it works.

Click here for the answer, once you‚Äôve thought about it!

Answer: This answer will be a little pedantic. As noted above, LLaMA 3-70B is initially trained with sequences of length 4K, so the batch size of 4M tokens gives us a sequence batch size of 1024. That means we can only really do pure data parallelism/FSDP up to 1024 chips because that‚Äôs how many sequences we have to do data parallelism over. So the answer in the simple sense of ‚Äúfully data parallelism with no extra communication‚Äù is no. The next question will answer a slightly less pedantic version of this.

Question: Let‚Äôs relax the requirement of not doing any sequence sharding. If we allow ourselves to do FSDP over both the batch and sequence axes, can we train LLaMA 3-70B with only FSDP on 8960 chips?

Click here for the answer, once you‚Äôve thought about it!

Answer: Now that we‚Äôre allowing ourselves to do sequence/context parallelism as well, we can scale up way more. First let‚Äôs calculate our per-device batch size. If we do 8960-way FSDP, we end with a per-TPU batch size of 4 * 1024 * 1024 / 8960 = 468 tokens. We know from the previous section that we become ICI-bound by FSDP when \(\text{per device batch size} < 2550 / n_\text{axes}\). Since we can dedicate 3 axes here with a full 3D pod, this would give us a lower bound of 850, which we‚Äôre well below. So the answer is no, even with 3 axes. We would be solidly communication-bound.

Question: Now let‚Äôs look at mixed tensor parallelism and FSDP. Does there exist some combination that lets us remain compute-bound? What amount of FSDP and tensor parallelism should we do if so?

Click here for the answer, once you‚Äôve thought about it!

Answer: First let‚Äôs check to see if this will even fit. We know that we‚Äôll be comms-bound if our per-chip batch size is less than \(2 \cdot 2550^2 / F = 453\). As we saw above, we‚Äôre slightly above this. So that‚Äôs great! Now to pick the optimal amount of FSDP, we can use the formula

\[X_{opt} = \sqrt{\frac{2BN}{F}} = \sqrt{\frac{2 \cdot 4.19e6 \cdot 8960}{28672}} = 1618\]

Rounding to a reasonable multiple of 2, that gives us roughly 2048-way FSDP and 4-way model parallelism. That should work well!

Takeaways: We can train LLaMA-3 with a 4M token batch size on a full TPU v5p pod with a mixture of data parallelism (1024-way), sequence parallelism (2-way), and tensor parallelism (4-way) without being communication-bound. We will be comms-bound if we try to do pure FSDP or FSDP + sequence parallelism. The equations we‚Äôve cooked up in the previous section are very practical.
